# Tech Design — Atelier Code Projects

Source brief: `docs/ideas/projects.md`. Implementation plan: `docs/plans/projects.md`.

## Scope

Projects are Atelier Code-owned records that name one workstation directory and
group Sessions. This design covers:

- A `projects` table and an optional `sessions.project_id` association.
- A `internal/project` domain package owning Project validation, IDs,
  timestamps, archive state, and directory rules.
- API endpoints for Project create, list, show, rename, archive, unarchive,
  and project-scoped Session listing and creation.
- CLI `atc projects` commands and `atc sessions start --project`.
- Admin web UI Project management (phase 2).
- Cohesion groundwork applied across the codebase before the feature lands
  (phase 0): shared public-ID generation, shared Go wire types, working
  directory validation on all Session starts, and store error renames.

Out of scope (per brief non-goals): Project deletion, directory changes after
creation, name/directory uniqueness, search, templates, multi-root Projects,
cross-workstation identity, app-client (Swift) work, and an `/api/fs`-backed
directory picker in the web create-Project form.

## Domain Model

`internal/project` owns the Project domain:

```go
type Project struct {
    ID         string     // prj_-prefixed opaque public ID
    Name       string     // required, renameable, not unique
    WorkingDir string     // required, absolute, fixed after creation
    CreatedAt  time.Time
    UpdatedAt  time.Time
    ArchivedAt *time.Time // nil = active
}
```

Rules (all enforced in the domain, not in clients):

- `Name` MUST be non-empty after trimming whitespace. No other constraint.
- `WorkingDir` MUST be absolute, MUST exist, and MUST be a directory at
  creation time. It is stored cleaned (`filepath.Clean`) and never changes.
- IDs are generated by the shared `internal/publicid` generator with prefix
  `prj_` (128 bits of `crypto/rand`, lowercase base32hex, no padding —
  identical scheme to `ses_`).
- Archive and unarchive are idempotent: archiving an archived Project and
  unarchiving an active Project are no-ops that return the current record.
- Archived Projects MUST NOT start new Sessions. All other reads (show, list
  with `includeArchived`, project-scoped Session listing) keep working.

Sentinel errors:

| Error                  | Meaning                                              |
| ---------------------- | ---------------------------------------------------- |
| `ErrProjectNotFound`   | ID does not exist.                                   |
| `ErrInvalidProject`    | Name (or other record field) fails validation.       |
| `ErrInvalidWorkingDir` | Path is relative, missing, or not a directory. The message states which check failed. |
| `ErrProjectArchived`   | A Session start referenced an archived Project.      |

`project.ValidateWorkingDir(path string) error` is the single directory rule:
absolute + `os.Stat` exists + `IsDir`. It is used by Project creation, by
project-scoped Session start revalidation, and by unscoped Session start
validation (see Session Integration).

`project.Service` (constructor `NewService(st *store.Store, logger
*slog.Logger)`) exposes `Create`, `Get`, `List(includeArchived bool)`,
`Rename`, `Archive`, `Unarchive`, and `ResolveForStart(ctx, id)` — the one
call the Session domain uses, which loads the Project, rejects archived
Projects (`ErrProjectArchived`), and revalidates the working directory
(`ErrInvalidWorkingDir`).

## Store

One goose migration `internal/store/migrations/0002_projects.sql`:

```sql
CREATE TABLE projects (
    id TEXT PRIMARY KEY NOT NULL CHECK (id <> ''),
    name TEXT NOT NULL CHECK (name <> ''),
    working_dir TEXT NOT NULL CHECK (working_dir <> ''),
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL,
    archived_at TEXT
);
CREATE INDEX projects_created_at_idx ON projects(created_at DESC);
CREATE INDEX projects_archived_at_idx ON projects(archived_at);

ALTER TABLE sessions ADD COLUMN project_id TEXT REFERENCES projects(id);
CREATE INDEX sessions_project_id_idx ON sessions(project_id);
```

Store conventions carry over exactly: fixed-width `timestampLayout` TEXT
timestamps (load-bearing for lexicographic `ORDER BY`), `COALESCE`-based
idempotent archive, incremental `clauses`/`args` filter building, and
`ORDER BY created_at DESC, rowid DESC` for newest-first lists.

Store API additions:

- `store.Project`, `store.CreateProjectInput{ID, Name, WorkingDir}`,
  `store.ProjectListFilter{IncludeArchived bool}`.
- `CreateProject`, `GetProject`, `ListProjects`, `RenameProject`,
  `ArchiveProject` (COALESCE set-once), `UnarchiveProject` (set NULL;
  `updated_at` bumps only when state actually changed).
- `store.CreateSessionInput` gains `ProjectID string` (empty means unscoped,
  stored as NULL via `nullableString`).
- Session reads (`Get`, `List`) LEFT JOIN `projects` and hydrate an optional
  nested value on `store.Session`:

```go
type SessionProject struct {
    ID         string
    Name       string
    WorkingDir string
    ArchivedAt *time.Time
}
// store.Session gains: Project *SessionProject
```

- `store.ListFilter` gains `ProjectID string`; when set, the list is
  restricted to that Project's Sessions.
- Error rename: `store.ErrNotFound` becomes `store.ErrSessionNotFound`;
  `store.ErrProjectNotFound` is added. The store package doc comment changes
  from "session metadata" to "Atelier Code-owned state".

Projects are never deleted (non-goal), so the FK needs no ON DELETE action.

## Session Integration

`session.StartInput` gains `ProjectID string`. `session.Session` gains
`ProjectID string` and `Project *ProjectRef` (ID, name, working dir,
archived-at), populated from the store join. `session.Service` gains a
`ProjectResolver` dependency, following the `ActionLoader` precedent so tests
can fake it:

```go
type ProjectResolver interface {
    ResolveForStart(ctx context.Context, id string) (project.Project, error)
}
```

Start behavior:

- `ProjectID` empty: current behavior, plus `WorkingDir` MUST now pass
  `project.ValidateWorkingDir` (absolute, exists, directory) instead of only
  non-empty. This applies to ALL Session starts, not only project-scoped
  ones — one rule, enforced once, inherited by every client.
- `ProjectID` set: `WorkingDir` MUST be empty (the API rejects the
  combination before the domain sees it). The service resolves the Project
  via `ResolveForStart`, inherits its working directory, and records both
  `project_id` and the inherited directory as the Session's `working_dir`
  snapshot. Archived Project → `ErrProjectArchived`. Vanished/invalid
  directory → `ErrInvalidWorkingDir`.
- Everything else about Session start (Action, Environment, name, prompt,
  params, launch, failure recording) is unchanged.

`session.Service.List` gains a project filter (threaded to
`store.ListFilter.ProjectID`) used only by the project-scoped listing route.

## API Contract

New routes (mounted under `/api`, behind the existing auth boundary):

```
POST  /projects
GET   /projects                       ?includeArchived=<bool>
GET   /projects/{id}
PATCH /projects/{id}
POST  /projects/{id}/archive
POST  /projects/{id}/unarchive
GET   /projects/{id}/sessions        ?includeArchived=<bool>&status=<status>
```

There is no `?projectId=` filter on `GET /sessions`;
`GET /projects/{id}/sessions` is the single project-scoped listing route. It
returns the same `{"sessions": [...]}` shape as `GET /sessions`, supports the
same `includeArchived` and `status` params, and returns 404
`project_not_found` for an unknown Project — a clear error instead of a
silently empty list.

Project wire shape (list and detail are identical; Project detail includes
only record fields, never inline Sessions):

```json
{
  "id": "prj_0abc...",
  "name": "Atelier Code",
  "workingDir": "/home/jeremy/Projects/atelier-code",
  "createdAt": "2026-07-07T15:04:05.123456789Z",
  "updatedAt": "2026-07-07T15:04:05.123456789Z",
  "archivedAt": "2026-07-08T09:00:00Z"
}
```

`archivedAt` is omitted when active. `GET /projects` returns
`{"projects": [...]}` newest-created-first, hiding archived unless
`includeArchived=true`.

Request bodies:

- `POST /projects`: `{"name": "...", "workingDir": "..."}`. Both required.
  Returns 201 with the full record (matching `POST /actions`).
- `PATCH /projects/{id}`: `{"name": "..."}`. Decoded with
  `DisallowUnknownFields` — any other field, including `workingDir`, is a 400.
  This is the codebase's first PATCH route and its first strict decode; the
  strictness is deliberate so a client cannot believe it changed the working
  directory. Returns 200 with the full record.
- `POST .../archive`, `POST .../unarchive`: no body. Return 200 with the full
  record; idempotent.

`POST /sessions/start` gains optional `projectId`. Validation order in the
handler: `action` required → exactly one of `workingDir` / `projectId`
required (both or neither is a 400 `invalid_request` with a message naming
the conflict) → domain. Session list/detail responses gain a nested object,
present only for Project Sessions:

```json
"project": {
  "id": "prj_0abc...",
  "name": "Atelier Code",
  "workingDir": "/home/jeremy/Projects/atelier-code",
  "archivedAt": "2026-07-08T09:00:00Z"
}
```

Errors use the existing envelope and a `writeProjectError` mapper parallel to
`writeSessionError`:

| Code                  | HTTP | When                                                       |
| --------------------- | ---- | ---------------------------------------------------------- |
| `invalid_request`     | 400  | Bad JSON, missing/blank `name`, unknown PATCH field, `workingDir`+`projectId` conflict, bad `includeArchived`. |
| `invalid_working_dir` | 400  | Directory is relative, missing, or not a directory — at Project create, any Session start, or project-scoped start revalidation. |
| `project_not_found`   | 404  | Unknown `{id}` on any `/projects/...` route.               |
| `project_not_found`   | 400  | Unknown `projectId` on `/sessions/start` (mirrors `unknown_action`). |
| `project_archived`    | 409  | Session start referencing an archived Project (mirrors `action_disabled`). |

API wire structs live in `internal/api` and are exported (see Shared Wire
Types) with the standard `formatTime`/`formatOptionalTime` timestamp
handling.

## CLI

New `projectsCommand` family registered in `root.go`, all supporting
`-o, --output text|json` where they return records (JSON mode is the raw API
body, per convention):

```
atc projects create --name <name> --dir <path>
atc projects list [--include-archived]
atc projects show <id>
atc projects rename <id> <name>
atc projects archive <id>
atc projects unarchive <id>
```

- `create`: `--name` and `--dir` required. `--dir` accepts relative paths
  (resolved against the CLI's cwd), `~`, and `~/...` (expanded via the user's
  home directory) before calling the API. `~user` and environment variables
  are NOT expanded in v1. Text output prints `<id>\t<name>`.
- `list`: text output prints one tab-separated row per Project:
  `id  name  workingDir  archivedAt` (empty last column when active).
- `show`: text output prints `key\tvalue` lines like `sessions show`.
- `rename`: uses a new `apiClient.patch` method (mirroring `post`). Text
  output prints `<id>\t<name>`.
- `archive`/`unarchive`: reuse the `sessionActionCommand`-style factory
  (generalized to take an endpoint builder); text output prints the id.

Session command changes:

- `atc sessions start --project <id>`: mutually exclusive with `--dir`
  (cobra `MarkFlagsMutuallyExclusive`). When `--project` is set the cwd
  default for `--dir` does NOT apply and no `workingDir` is sent.
- `atc sessions list --project <id>`: calls
  `GET /projects/{id}/sessions`, combinable with `--status` and
  `--include-archived`. (Addition beyond the brief: the API supports
  project-scoped listing, so the CLI exposes it.)
- `sessions show` text output prints `project` fields when present; the
  `sessions list` text columns are unchanged.

## Admin Web UI (Phase 2)

New sidebar entry **Projects** and two routes following the existing
hand-rolled CSS + `$state`/`onMount` patterns:

- `/projects` — list with an explicit include-archived toggle, newest-first,
  and a create modal (`ProjectEditor` component following the `ActionEditor`
  prop contract: name text input, working directory text input). The
  directory input is a plain text input; an `/api/fs`-backed browse picker
  is out of scope for this version (deferred).
- `/projects/[id]` — Project detail: record fields, rename (same editor in
  rename mode, name only), archive/unarchive buttons, the Project's Sessions
  via `GET /projects/{id}/sessions` with an include-archived toggle
  (archived hidden by default), and a **Start Session** form: Action select
  (enabled Actions from `GET /actions`), Environment select
  (`GET /environments`), optional Session name and prompt, and param inputs
  rendered from the selected Action's params spec. Submits
  `POST /sessions/start` with `projectId` and no `workingDir`.

Supporting changes:

- `api.ts` gains `Project` types and `listProjects`, `getProject`,
  `createProject`, `renameProject`, `archiveProject`, `unarchiveProject`,
  `listProjectSessions`; `SessionListItem`/`SessionDetail` gain the optional
  `project` object; `listSessions` gains `includeArchived`.
- Sessions page retrofit (minimal): each Session row shows its Project name
  linking to `/projects/[id]`, and the page gains the include-archived
  control the API already supports.
- The reference/Try-it console gains the Project endpoints: `endpoints.ts`'s
  method union widens to include `PATCH` and the Try-it request builder
  handles it like `POST`.
- The copy-pasted inline error-banner markup is extracted into a shared
  `ErrorBanner` component and adopted by the new pages and, opportunistically,
  the existing ones.

## Cohesion Groundwork (Phase 0)

Changes that make the codebase better independent of Projects and land
first, each shippable alone:

1. **`internal/publicid`**: shared `New(prefix string) (string, error)`
   generator; `internal/session/id.go` becomes a call with `ses_`, Projects
   use `prj_`.
2. **Shared Go wire types**: the `internal/api` session wire structs
   (`SessionListItem`, `SessionDetail`, list/detail response envelopes) are
   exported and the CLI imports them instead of redeclaring
   byte-for-byte copies. The CLI and server ship in one binary, so there is
   no dependency-weight concern. TypeScript types in `api.ts` remain
   hand-written (accepted cost).
3. **Store error rename**: `store.ErrNotFound` → `store.ErrSessionNotFound`;
   store package doc comment updated to cover all Atelier Code-owned state.
4. **Working directory validation on all Session starts**: `session.Start`
   validates absolute/exists/is-directory instead of only non-empty.
   Behavior change: a bad directory now fails fast with 400
   `invalid_working_dir` instead of failing at multiplexer launch with 502
   `launch_failed`.

## Test Plan

Follow the prevailing scenario-based stdlib-only patterns per layer:

- `internal/store`: migration schema assertions (`sqlite_master`, FK pragma),
  Project CRUD round-trips, idempotent archive/unarchive, newest-first
  ordering with rowid tie-break, session↔project join hydration,
  `ProjectID` list filtering, NULL `project_id` compatibility for pre-existing
  rows.
- `internal/project`: validation matrix for name and working directory
  (relative, missing, file-not-dir, symlinked dir), `ResolveForStart` against
  active/archived/deleted-directory Projects, rename, archive semantics.
- `internal/session`: project-scoped start happy path (inherited dir
  snapshot, `project_id` persisted), archived-project rejection, revalidation
  failure, unscoped starts unchanged plus new directory validation, fake
  `ProjectResolver`.
- `internal/api` (`projects_test.go` + additions to `sessions_test.go`):
  route-by-route success shapes, full error-code table above, PATCH
  unknown-field rejection, `workingDir`+`projectId` conflict, nested
  `project` object presence/absence, project-scoped session listing including
  404 and filter params, nil-service guard.
- `cli` (`projects_test.go`): Unix-socket end-to-end per command, `--dir`
  relative/`~` resolution, text and raw-JSON output assertions,
  `--project`/`--dir` mutual exclusion, `sessions list --project` endpoint
  shape, `apiClient.patch`.
- Full validation: `go test ./...`, `cd web && npm run build` (phase 2), and
  `git diff --check`.
