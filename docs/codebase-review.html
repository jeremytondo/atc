P1 Findings

Fix these before the next major feature wave. They are foundational reliability, security, and maintainability gaps.

P1-01

CI covers only the Linux server path

The shipped surfaces do not have a shared automated quality gate. Go formatting, frontend type checking, Swift app tests, and macOS behavior can regress while CI stays green.

Review evidence and recommendation

Evidence

  • Existing CI runs only the server workflow on Ubuntu.
  • No workflow covers macos/, packages/, repo scripts, or root configuration.
  • Local checks already expose unformatted Go files, a macOS Go test failure, missing TypeScript types, and Svelte warnings.

Recommendation

  • Add server gates for gofmt, vet, Go tests, frontend checks, frontend tests, and embedded build.
  • Add a macOS/Xcode job for Swift package and app tests.
  • Use timeouts, concurrency cancellation, and clear warning ownership.
P1-02

Remote unauthenticated server bind fails open

An explicit non-loopback TCP bind without a token starts the service instead of failing closed. The warning also fires for every remote bind, even when auth is present.

Review evidence and recommendation

Evidence

  • server/internal/server/server.go logs the unauthenticated warning without checking cfg.AuthToken.
  • The exposed API can browse host paths, launch configured actions, inject terminal input, terminate sessions, and attach to PTYs.
  • The runtime cannot distinguish an intentional tailnet-only mode from an unsafe LAN bind.

Recommendation

  • Fail startup for non-loopback TCP binds with no token.
  • Require an explicit trusted-network override if that mode remains supported.
  • Test loopback/no-token, remote/token, remote/no-token, and explicit override behavior.
P1-03

HTTP JSON requests have no size limit or body deadline

JSON handlers decode directly from request bodies. A client can hold handlers open with a slow body or send oversized input until decoding or allocation fails.

Review evidence and recommendation

Evidence

  • The server configures ReadHeaderTimeout, but not request-body controls for ordinary API routes.
  • decodeJSON reads directly from r.Body with no http.MaxBytesReader or trailing-value check.
  • The attach pre-resize buffer caps each frame but not aggregate retained input.

Recommendation

  • Use a shared decoder with endpoint-appropriate byte limits and stable errors.
  • Add body read deadlines or middleware without breaking WebSocket attaches.
  • Cap aggregate pre-resize attach input.
P1-04

Native bearer tokens are persisted in plain app preferences

Storing tokens in UserDefaults was a reasonable early vertical-slice choice. With durable multi-server settings and remote access, the deferral has reached its useful limit.

Review evidence and recommendation

Evidence

  • ConnectionRecord.token is encoded with the connection array under one UserDefaults key.
  • The app now has persistent connection records, distribution automation, and remote server support.
  • The browser admin UI has a separate storage decision and should not dictate native storage.

Recommendation

  • Keep names, URLs, identity, and order in UserDefaults.
  • Store each token as a Keychain item keyed by stable connection UUID.
  • Migrate plaintext tokens once after a verified Keychain write.
P1-05

The API contract is manually duplicated across implementations

Go, Swift, TypeScript, and docs each carry their own version of endpoint paths and wire models. The right first fix is lightweight shared fixtures, not heavy code generation.

Evidence

  • Go wire structs live in server/internal/api.
  • TypeScript repeats models and paths in server/web/src/lib/api.ts.
  • Swift repeats models in packages/AtelierCodeKit.
  • API-reference data is a fourth manual description and already has behavior drift.

Recommendation

  • Add canonical request/response fixtures.
  • Have each client validate those fixtures in its own tests.
  • Revisit OpenAPI or generated models only after fixtures hit real limits.

P2 Findings

These are real defects or structural weaknesses with contained current impact. They should be scheduled soon.

P2-01

Terminal registry presence is treated as a live connection

Retained terminal controllers preserve scrollback, but UI indicators treat dictionary membership as active attachment state. Add a derived active phase and use it for connection status and destructive-save warnings.

P2-02

Refresh ownership is duplicated

ConnectionRuntime owns real polling, while unused store poll loops remain. Overlapping refreshes can also clear loading state too early. Consolidate ownership and refresh runtimes concurrently.

P2-03

Frontend validation is below the Go and Swift surfaces

Add @types/node, svelte-check, a clean check command, focused tests, and runtime response validation through the API contract work.

P2-04

The Go suite is not portable to macOS

TestListSorting expects both AB and ab in one temp directory. Split comparator tests from filesystem integration tests.

P2-05

Active docs still define retired Cockpit-era system names

Classify docs as current, historical, or superseded. Update current docs to Atelier Code naming and add a lightweight check for retired executable identifiers outside archive paths.

P2-06

macOS tests pass with runtime warnings

Optional Pickers are hosted with invalid nil selections, and a reentrant NSTableView warning says it will become an assertion in the future.

P3 Findings

These are useful policy decisions and cleanup tasks, not reasons to pause current work by themselves.

P3-01

Decide and document the macOS compatibility and security floor

The Xcode project, Swift package, sandbox policy, ATS policy, and HTTP-token behavior need one current source of truth. The recommended default is HTTPS for non-loopback connections unless a trusted overlay override is explicit.

P3-02

Small readability and performance cleanup

Run and enforce gofmt, bound terminal outbound streams, avoid quadratic pending-output drains, handle corrupted URL records explicitly, and monitor the route-isolated terminal bundle size.

Stabilization Sequence

A practical order that makes the truth visible first, then closes the highest-risk boundaries.

1
Make the truth visible Add CI/check commands, fix the macOS Go test, restore gofmt cleanliness, fix Svelte and Picker warnings, and get current checks green.
2
Close trust-boundary gaps Fail closed for unauthenticated remote binds, bound HTTP bodies and lifetimes, and move native secrets to Keychain.
3
Protect the monorepo contract Add shared API fixtures and cross-client contract tests before adding more endpoints.
4
Fix lifecycle semantics Separate retained terminal history from active connection state and consolidate refresh/poll ownership.
5
Make documentation current Archive or update Cockpit-era plans and record the macOS compatibility/network policy.
6
Resume feature work Preserve the existing service boundaries and tests; they are the strongest part of the foundation.

Open Questions

These decisions shape the implementation details of the stabilization work.

Minimum macOS version

Use the oldest version actually supported by the chosen Ghostty package and APIs, not the developer machine's current patch release.

Remote trust model

Default to token-required for every non-loopback bind. If tailnet identity alone is supported, make it an explicit trusted-network override.

API contract mechanism

Start with shared fixtures plus runtime validation. Revisit OpenAPI only after the next meaningful contract expansion.

Browser token persistence

Decide whether the admin UI is a trusted-local tool or remotely exposed product surface before choosing localStorage, session storage, or cookies.
